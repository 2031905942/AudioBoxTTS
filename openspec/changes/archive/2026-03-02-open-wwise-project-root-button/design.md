## Context

当前“Wwise工程”配置项仅支持通过“选择Wwise工程文件”写入 `.wproj` 文件路径，但缺少一键跳转到该工程目录的入口。用户需要频繁在资源管理器中手动定位该目录，效率低且容易定位错误。

该变更是单模块 UI 增强：位于 `Source/UI/Interface/ProjectInterface/project_tab_window.py`，需要复用现有的 Wwise 路径配置状态，并在按钮点击时调用系统能力打开目录。

## Goals / Non-Goals

**Goals:**
- 在“选择Wwise工程文件”按钮右侧新增“打开Wwise工程根目录”按钮。
- 仅在已配置有效 `.wproj` 路径时允许点击。
- 点击后打开 `.wproj` 所在目录；若目录不可用，给出提示并保持程序稳定。

**Non-Goals:**
- 不调整 Wwise 路径配置的数据结构与持久化格式。
- 不修改 SoundBank 同步逻辑或其他项目页功能。
- 不新增跨平台行为分支（本次以 Windows 文件资源管理器行为为主）。

## Decisions

- 决策 1：复用当前页面已有的 `.wproj` 路径状态作为目录来源。
  - 原因：避免新增配置字段，减少状态不一致风险。
  - 备选：单独维护“工程目录”配置；否决，因冗余且易与 `.wproj` 脱节。

- 决策 2：目录路径通过 `Path(wproj).parent` 计算。
  - 原因：语义清晰，直接表达“工程根目录 = 工程文件所在目录”。
  - 备选：字符串裁剪；否决，可读性与健壮性较差。

- 决策 3：使用 Windows 原生方式打开目录（如 `os.startfile`）。
  - 原因：与当前运行环境一致，用户体验直接。
  - 备选：`subprocess` 调起 `explorer.exe`；保留为降级方案，不作为首选。

- 决策 4：按钮启用态随 `.wproj` 选择状态联动，并在点击时再次校验目录存在性。
  - 原因：同时覆盖“正常路径”和“配置过期/目录被删除”场景。

## Risks / Trade-offs

- [路径已配置但目录被移动/删除] → 点击时二次校验目录并弹出错误提示。
- [UI 布局空间变紧] → 复用现有行布局组件，保持与相邻控件同级对齐，避免引入新容器复杂度。
- [平台差异] → 本变更按 Windows 环境实现；未来跨平台时再在同一入口扩展分支。
